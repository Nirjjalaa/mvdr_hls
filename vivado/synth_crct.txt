Why changing Vivado clock is okay

In HLS, 10 ns was the target used during HLS scheduling.

But after exporting the IP, the actual FPGA clock comes from Vivado:

versal_cips_0/pl0_ref_clk → mvdr_top_axi_0/ap_clk

If you run the hardware slower, the circuit gets more time per cycle.

100 MHz = 10 ns per cycle
50 MHz  = 20 ns per cycle

So changing to 50 MHz is not wrong. It is actually safer for timing.

Will 50 MHz definitely work?

I need to be honest: I cannot guarantee before implementation finishes.

But based on your note:

Worst slack = -1.936 ns at around 100 MHz

that means the path needs roughly:

10 ns + 1.936 ns = 11.936 ns

At 50 MHz, Vivado gives:

20 ns

So 50 MHz gives much more timing margin.

Therefore:

100 MHz: likely timing problem
50 MHz: very likely to pass timing

But the only final proof is:

Run Implementation → Timing Summary → WNS >= 0

If WNS is positive or zero after implementation, then timing is met.
